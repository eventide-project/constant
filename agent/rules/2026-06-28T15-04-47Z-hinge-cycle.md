# The initial-implementation hinge cycle: actuation → assertion → controls → implementation

Building the initial implementation of an outcome runs a cycle of **four hinges**, each handed to the human separately and accepted before moving to the next:

1. **Actuation** — the invocation of the unit under test (its efferent shape: name, arguments, arity, return).
2. **Assertion** — what is asserted about the result (value-equality vs. identity vs. a raised error; what the explaining variable is compared against).
3. **Controls** — the example constants the test is built from (which `Controls::Constant` form, nested vs. top-level, what inner constants, Array vs. Hash seeding, literals vs. controls).
4. **Implementation** — the code that satisfies the test.

The first three concern the test (turn one); the fourth is the implementation (turn two). Only after the actuation, assertion, and controls are accepted does the AI assemble and show the turn-one test file (per the first-turn rule); then it proposes the implementation as the fourth hinge, and — once that is accepted — writes it, runs the suite, and shows the test with its result.

**The handoff mechanism at every hinge:** if there are genuinely distinct options, present them **as options**; if there are none, **prompt the human to accept** the single proposal. **In both cases, always offer a "Chat about this" option** — the dictate/chat escape. Never pass a hinge without the human's acceptance. This generalizes the actuation-gate options-or-chat rule: not every hinge has real alternatives, so do not manufacture fake options — but the origination escape ("Chat about this") is mandatory at every hinge regardless (limit A: the best choice is often one only the human can produce, and a closed proposal must never foreclose it).

**Why:** Each of the four is a real design hinge with its own failure mode — including the implementation, where solubility is judged (an awkward implementation reveals a bad cradle, and the fix then goes to the cradle, not the code). Bundling collapses separate deliberations into one judgment and puts the human in judge-mode on a fait accompli (the Law-1 mean-bias failure). Separating them — each with a mandatory Chat escape — keeps the human originating at every hinge, where the above-the-mean choice is introduced rather than ratified. The assertion and the controls especially had been smuggled in under the actuation's gate; the controls are a hinge in their own right (a non-discriminating example asserts nothing — e.g. a top-level constant can't distinguish a full name from a final segment).

**How to apply:** Run the four hinges in order. At each: options if there are genuine ones, else prompt to accept, always with "Chat about this." The per-hinge presentation: the **actuation hinge** shows the actuation inside its nested folder-mirroring `context` blocks; the **assertion hinge** shows the actuation + assertion; the **controls hinge** shows the controls + actuation + assertion (the full test body, no contexts). The actuation is the **anchor shown at the downstream hinges** (a hinge is meaningful only in relation to the call), and the nested contexts are shown once at hinge 1, not repeated thereafter. See the actuation-hinge-shows-contexts, assertion-hinge-shows-actuation, and controls-hinge-shows-actuation rules. Assemble and show the turn-one test after the three test hinges; propose the implementation, then (on acceptance) write it and show the test + run result for the fourth. This is the **initial**-implementation cycle for one outcome; looping to further outcomes extends the implementation and re-enters at the relevant hinges. Related: the first-turn rule (the three test hinges produce turn one), the human-in-the-loop rule (actuation and solubility were the original two hinges; the implementation hinge is that solubility judgment made an explicit acceptance), the actuation-gate options-or-chat rule (the mechanism, here generalized to options-or-accept), the one-outcome-at-a-time rule, and the show-the-test-after-implementing rule.
