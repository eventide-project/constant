# Drop an outcome whose test is green-on-arrival — it drives no design

In this TDD-as-design approach, a test earns its place by **driving** an implementation that did not exist — moving red → green. An outcome whose test passes the moment it is written ("green-on-arrival") drives nothing: the behavior already exists, so the test only *confirms* an affordance rather than *designing* one. Drop it.

**The tell is a no-op implementation hinge.** In the four-hinge cycle (actuation → assertion → controls → implementation), if hinge 4 degenerates to "nothing to write — the existing code already satisfies this," the outcome is not a genuine design step. That degeneration is the signal to drop the outcome, not to add a confirming test.

**Why:** TDD here is a design tool, not a coverage tool (see the TDD-as-design and human-in-the-loop rules: *exit on solubility, not coverage*). A green-on-arrival test designs nothing and dilutes the suite's signal — it reads as a driven outcome when it isn't. Such a test usually appears because the implementation ran **ahead** of its driving test (a default value, a threaded keyword, a guard written incidentally while satisfying an earlier outcome), so the behavior was already present when its "own" outcome came up. The discipline is to notice that and not manufacture a test to cover it after the fact.

**How to apply:** When an outcome's implementation hinge is a no-op (the test would be green-on-arrival), drop the outcome rather than keeping the confirming test. If the behavior genuinely needs locking as a public-contract regression guard, that is a deliberate, separate decision the human makes — it is not the default, and it is not a TDD design step. Related: the TDD-as-design-tool rule, the human-in-the-loop rule (exit on solubility), the one-outcome-at-a-time rule, and the hinge-cycle rule (hinge 4).
