# Never ask whether to start a task test-first — test-first is the default

Test-first is the standing default for starting any task here. Do not ask "shall I start it test-first?" or otherwise put the test-first decision to the human — there is nothing to deliberate at that point; the answer is always yes unless told otherwise. If the human wants a task started some other way, they will give a different instruction.

**Why:** Asking confirms a settled default and spends a turn on a non-decision. The human's attention belongs at the hinges — the efferent call and the solubility verdict — not on whether to follow the project's own method (see the human-in-the-loop-TDD and TDD-as-design-tool rules). Re-asking the standing default is ceremony, the same failure the TDD rules retire.

**How to apply:** When the next task is clear, begin it test-first without asking. Offer the next-task choice if useful, but never frame it as a test-first/not question. Deviate from test-first only on an explicit instruction to do so.
